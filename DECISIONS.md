# 决策日志 (Decision Log)

记录产品方向与功能的关键决策，按时间与逻辑顺序排列。

---

1. **从 Debate 机器人起步**  
   最初的想法链路是做一个 Debate 机器人，用 AI 进行辩论式对话。

2. **Debate 机器人缺乏市场切入点**  
   判断 Debate 机器人获取用户的门槛和阈值太高，难以找到清晰的市场切入点。

3. **转向「基于产品的追问与发散」**  
   考虑做一个围绕具体产品进行追问和发散的工具：既有更好的切入点，又能体现 AI Agent 的追问与推理能力。

4. **用思维导图承载 AI 追问**  
   决定用思维导图形式呈现结构，由 AI 在节点上发起追问，把「产品构想」拆成可逐项回答的问题。

5. **思维导图落地：追问 + 回答 + 再追问**  
   思维导图实现后，已具备基本流程：用户回答节点 → AI 可基于回答再追问，形成多轮、可扩展的对话。

6. **追问改为用户主动触发**  
   追问不自动全量生成，而是提供选项让用户点击后再由 AI 追问；更可控、更自定义，避免信息过载。

7. **加入 Tips：用户「不知道怎么写」时的帮助**  
   在追问与回答闭环完成后，发现用户常遇到「有问题但不知道如何答」的情况，因此加入 Tips 功能作为辅助。

8. **Tips 既可附属也可作为答案，且可多条并存**  
   Tips 既可作补充说明（附属），也可直接选为节点答案；支持多条 Tips 并存，用户按需选择或组合。

9. **Agent Skills：预置场景 + 导入 Markdown 优化回答**  
   已实现：在项目根目录 `skills/` 下放置 Markdown 文件（如 `product_bp.md`），每个文件为一个「场景/技能」；立项与脑图阶段的所有内容类 AI 调用会将该技能正文注入为 system 上下文，从而优化表述与专业度。前端在立项页提供「场景/技能」下拉框，选定后随 draft 与 from-draft 传给后端；项目保存 `skill_id`，后续追问、Tips、融合文档均沿用该技能。用户自导入：可将自写 .md 放入 `skills/` 目录即可使用。

10. **极速模型与思考推理模型的平衡**  
   现象：选用极速模型时响应快、交互顺畅，但回答质量（尤其是澄清本质、生成追问、融合文档）会受损；选用思考推理模型时质量更好，但响应慢，交互体验差。  
   平衡方案：**取名用极速、内容用推理**——节点短标题（取名）等轻量调用走极速模型，保证响应快；澄清本质、初题、脑图结构、追问、Tips 候选、回答判定、融合文档等「内容」类调用走推理模型，保证质量。已落地：后端 `ai_client.py` 支持 `AI_MODEL_FAST` / `AI_MODEL_REASONING` 两个可选环境变量，不填则均回退到 `AI_MODEL`。

11. **超级 Agent 触发 429 的应对方案**  
   现象：超级 Agent 在展开/收敛阶段会短时间发起大量 LLM 请求（回答建议、提交回答、追问等），单端点单 Key 容易触发 429 Too Many Requests。  
   方案：**多端点轮询 + 429 重试退避**。（1）支持多组 API：`AI_API_BASE` + `AI_API_KEY`、`AI_API_BASE_2` + `AI_API_KEY_2`、`AI_API_BASE_3` + `AI_API_KEY_3`；每次调用通过 `_next_endpoint()` 轮询到不同端点，将请求分散到多 Key/多服务商，降低单点限流概率。（2）若某次请求返回 429，先等待退避时间（第 1 次 3 秒、第 2 次 6 秒），再重试；重试时因轮询会换端点，往往命中未限流的 Key。最多重试 3 次，仍 429 则抛出 `RuntimeError("429 Too Many Requests after retries")`。  
   使用建议：配置 2～3 组 Base+Key（可同服务商多 Key，或不同服务商各一），见 `.env.example` 中「多模型分担压力」说明。
