# 熵导图 · 搜索检索 + 多模态输出 可行性计划

在**不改变现有基础功能**的前提下，为产品增加「融合搜索/网站检索」与「多模态输出」能力，提升可用性与表现力。本文档从场景、技术选型、接入点与分阶段落地做计划。

---

## 一、搜索 / 网站检索能力

### 1.1 可用的使用场景

| 场景 | 说明 | 可用性 |
|------|------|--------|
| **立项对话** | 用户说「做一个 AI 写作工具」时，检索当前市场产品、融资动态，辅助澄清本质或追问 | 高：对话上下文可注入检索摘要，不改变对话流程 |
| **节点回答 / Tips** | 回答或生成 Tips 时，按「问题 + 项目主题」查网页/文档，给出有依据的参考 | 高：在现有 `node_answer_judge_and_followups`、Tips 候选、直接回答前加一步检索即可 |
| **融合文档** | 生成最终方案时，为「市场分析」「技术选型」等段落引用来源（链接 + 摘要） | 中：需在 merge 接口中增加「可选引用块」与前端展示 |
| **技能/知识库** | 除本地 `skills/*.md` 外，按主题拉取外部资料（如竞品页、行业报告摘要）作为上下文 | 中：与现有 skill_content 并联，检索结果拼进 system 或 context |

### 1.2 技术选型（可选）

- **网页/通用搜索**  
  - **Serper / Tavily / Bing API**：关键词搜索 → 返回标题、摘要、URL；适合「立项 + 节点回答 + 融合引用」。  
  - **自建爬虫 + 摘要**：可控、可存历史，但维护成本高，建议作为二期。
- **文档检索（RAG）**  
  - 已有「上传文档 → 解析为 idea_text」；可扩展为：解析后做 chunk + 向量存储，回答/Tips 时按问题检索相关片段，注入 LLM。  
  - 技术：本地用 `sentence-transformers` + 轻量向量库，或接云向量库（如 OpenAI Embedding + 简单存储）。

### 1.3 接入点与可用性小结

- **接入点**  
  - `AIClient` 新增：`search_web(query, max_results=5)` → 返回 `[{title, snippet, url}]`。  
  - 在以下调用前按需调用检索，把结果拼进 prompt：  
    - 立项：`draft_append_message` 的 system 或最后一轮前插入「近期检索摘要」。  
    - 节点：`node_answer_judge_and_followups`、Tips 候选、`generate_node_answer` 等，在问题/项目主题上做一次 search，把摘要作为 context。  
  - 融合：`merge_project_doc` 可接受「可选引用列表」；由调用方在 merge 前对部分 section 做检索，传入引用，模型在文档中写「参考：…」。
- **可用性**  
  - 逻辑上全部是「在现有文本流程前/中加一步检索 + 拼 context」，不改变现有回答/追问/Tips/融合的输入输出契约。  
  - 需要：API Key（Serper/Tavily 等）或自建检索服务；若不做 RAG，仅网页搜索则无向量库依赖。

## 二、多模态输出能力

### 2.1 可用的使用场景

| 场景 | 说明 | 可用性 |
|------|------|--------|
| **节点附图** | 某节点（如「产品示意图」「架构图」）由 AI 生成或用户上传一张图，节点展示缩略图 + 大图预览 | 高：节点模型加 `image_url` 或 `attachment_id`，前端按需渲染 |
| **融合文档内图表** | 融合后的方案文档中含「架构图」「流程图」「竞品对比表」等，以图片或表格形式输出 | 中：若模型只出文本，可用「描述 + 前端图表库」生成；若接口支持 image out，可直接嵌图 |
| **立项/回答中的示意图** | 对话或回答中返回「一张示意图」（如产品形态草图、用户旅程），需模型或单独图像接口支持 | 中高：依赖多模态模型（image out）或 DALL·E/Midjourney 等 API |

### 2.2 技术选型（可选）

- **模型侧**  
  - **Image out**：若使用支持「生成图片」的接口（如 OpenAI image generation、或部分国产多模态模型），可在指定节点或融合阶段请求一张图，存 URL 或 base64，前端展示。  
  - **仅文本**：用「结构化描述」+ 前端图表库（如 Mermaid、ECharts）把描述转成图，无需图像 API。
- **存储**  
  - 图片可存本地（如 `static/uploads/` 或项目下 `media/`）+ DB 记路径；或存 OSS，DB 记 URL。
- **前端**  
  - 节点卡片：`<img>` 或背景图；融合文档区域：支持 Markdown 图片、或内嵌图表组件。

### 2.3 接入点与可用性小结

- **接入点**  
  - **Node 模型**：增加可选字段 `image_url: Optional[str]`（或 `attachment_id` 关联媒体表）。  
  - **回答/Tips 流程**：不必须改；若要做「该节点回答是一张图」，可在保存回答时把「图片 URL」写入节点或 NodeAnswer 的扩展字段，前端根据类型渲染。  
  - **融合文档**：  
    - 方案 A：merge 接口返回 `{ markdown: string, images?: [{ section_id, url }] }`，前端在对应段落嵌图。  
    - 方案 B：markdown 里含 `![desc](url)`，图片 URL 由后端在 merge 时生成或引用已存图。
- **可用性**  
  - 节点附图、融合文档插图：与现有「纯文本」兼容，不做多模态时字段为空即可。  
  - 对话/回答中直接出图：依赖多模态或图像 API，可作为「增强能力」分阶段上。

## 三、融合搜索 + 多模态的协同可用性

- **检索为多模态提供依据**：例如「竞品对比」节点，先检索 3 个竞品信息，再生成文字 + 一张对比表/图，表/图内容有据可查。  
- **多模态为检索结果增可读性**：检索到市场数据后，不仅写进正文，还可生成一张简单图表（如柱状图），提升融合文档质量。  
- **实现顺序建议**：先做「检索 → 文本增强」（立项、节点、融合），再做「多模态输出」（节点图、融合文档图表），最后做「检索 + 多模态」组合（如检索后生成图/表）。

---

## 四、分阶段落地建议（保持现有功能不变）

| 阶段 | 内容 | 依赖 | 风险/注意 |
|------|------|------|-----------|
| **P0 检索接入** | 引入 Serper/Tavily 等 API，在 AIClient 中实现 `search_web`；在 1～2 个调用点（如 Tips 候选或节点直接回答）注入检索摘要，观察效果 | 搜索 API Key | 延迟与 token 增加；可做开关「仅当配置了 SEARCH_API_KEY 时启用」 |
| **P1 检索扩展** | 立项对话、融合文档引用接入检索；可选：上传文档的 RAG 检索 | 同 P0；若 RAG 则需向量库/Embedding | 立项对话需控制 context 长度 |
| **P2 多模态-节点** | Node 支持可选 `image_url`；提供「生成该节点示意图」的入口（如右键或节点内按钮），调用图像接口并落库，前端展示 | 图像生成 API 或 Mermaid 等图表库 | 成本与审核（若用公共图像 API） |
| **P3 多模态-融合** | 融合文档支持插图/图表（Markdown 图片或前端图表组件）；可选：由模型输出「图表描述」再由前端渲染 | 同 P2 | 与现有 merge 文本格式兼容 |
| **P4 检索+多模态** | 在「竞品/市场/技术」类节点或段落，先检索再生成文字+图/表，并注明来源 | 依赖 P0～P3 | 产品上需明确「参考来源」的展示规范 |

## 六、总结

- **搜索/网站检索**：与现有流程兼容度高，在立项、节点回答/Tips、融合三处注入检索结果即可增强「有据可查」；可用性高，建议优先做检索接入与 1～2 个核心场景。  
- **多模态输出**：节点附图与融合文档插图均可设计为「可选扩展」，不破坏现有纯文本流程；可用性中高，建议在检索稳定后再做节点图与融合图表。  
- **两者结合**：检索提供依据、多模态提升呈现，适合在路线图后期做「检索 → 生成图/表」的闭环，并统一考虑引用与版权展示。

以上计划在**不改变现有基础功能**的前提下，按阶段加入能力即可；每步都可做成「有配置则启用、无配置则退化为当前行为」。
